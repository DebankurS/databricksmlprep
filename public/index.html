    AutoML vs Manual Training

    Choosing between Databricks AutoML and hand-coded training.

    CriteriaAutoMLManual Training
    Skill RequiredNo ML codeFull ML expertise
    SpeedMinutes to a modelHours to days
    CustomizationLimited (but editable notebooks)Unlimited
    ExplainabilitySHAP auto-generatedManual config
    Best ForBaselines, rapid prototypingProduction, custom architectures

    Spark ML Pipeline Stages

    Order and role of each stage type.

    Transformer (e.g. VectorAssembler, StringIndexerModel)
    Stateless or already fitted. Has .transform(df). Does not learn from data.
    Estimator (e.g. LogisticRegression, StringIndexer)
    Has .fit(df) → returns a Transformer (Model). Learns parameters from training data.
    Pipeline.fit(train) → PipelineModel
    Fits all Estimators in sequence. Call .transform(test) on the PipelineModel for inference.

    Hyperopt: Trials vs SparkTrials

    Parallelism and integration differences.

    Trials (Sequential)

    Runs HPO trials one at a time on the driver node. Use for small datasets or when Spark workers aren't needed. No cluster cost beyond driver.

    Best for: non-distributed models, quick experiments.
    SparkTrials (Parallel)

    Distributes trials as Spark tasks across workers. Use parallelism= to set concurrency. Auto-logs to MLflow on Databricks.

    Best for: scikit-learn/XGBoost HPO at scale, large search spaces.

    MLflow Registry Stages

    Model version lifecycle states.

    None → Staging
    New version under evaluation. Automated testing runs here. Not serving production traffic.
    Staging → Production
    Validated, serving live traffic. Load via `models:/Name/Production`. Only one version per stage at a time.
    → Archived
    Retired version. Kept for audit/rollback. Not actively served. Use Aliases in Unity Catalog instead of stages.

    Stages vs Aliases in Model Registry

    Unity Catalog deprecates stages; use aliases.

    Stages (Legacy)

    Fixed enum: None / Staging / Production / Archived. One version per stage. Deprecated in Unity Catalog model registry.

    Load: mlflow.pyfunc.load_model("models:/MyModel/Production")
    Aliases (Preferred in UC)

    User-defined names (champion, challenger, shadow) pointing to any version. Multiple aliases can coexist on different versions simultaneously.

    Load: mlflow.pyfunc.load_model("models:/catalog.schema.Model@champion")

    Drift Types & Detection

    Data Drift (Covariate Shift) — P(X) changes
    Input feature distributions shift. Detected via PSI, KL divergence on feature columns. May or may not degrade model.
    Concept Drift — P(Y|X) changes
    Feature-label relationship changes. Detected only via labeled production data. Requires retraining. Most impactful.
    Label Drift — P(Y) changes
    Target distribution shifts (e.g. churn rate changes from 5% to 20%). Monitor prediction distribution as proxy when labels delayed.

    PSI Thresholds for Drift

    Population Stability Index decision rules.

    PSI < 0.1 — Stable
    No significant distribution shift. Continue monitoring at normal frequency.
    PSI 0.1 – 0.25 — Moderate Shift
    Some change detected. Increase monitoring frequency and investigate data pipeline.
    PSI > 0.25 — Major Shift
    Significant distribution change. Trigger investigation and likely retraining pipeline.

    ML CI/CD Pipeline Stages

    CI — Continuous Integration

    Git push → run unit tests → run training job → log to MLflow → register model in Staging.

    Validates: code correctness, model performance meets threshold.
    CD — Continuous Delivery

    Staging model → automated validation job → if passes, promote alias to Production → downstream serving picks up new version.

    Validates: holdout test performance, latency, integration tests.
